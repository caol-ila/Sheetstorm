#### #101: copyWith Sentinel Pattern

**Status:** ✅ VERIFIED (previously completed)  
**Pattern:** Sentinel value (private const `_undefined`) in Freezed models to distinguish "not set" from `null`.

Example:
```dart
@Freezed
class UserModel with _$UserModel {
  const UserModel._();
  
  const factory UserModel({
    required String id,
    @Default(_undefined) String name,
  }) = _UserModel;
  
  static const _undefined = "___undefined___";
}
```

**Benefit:** Freezed's `copyWith()` can differentiate between "omitted" and "explicitly null".

### MS3 — Tuner Feature (In Progress)

#### Architecture Decisions Document

**Location:** `.squad/decisions/inbox/romanoff-tuner.md`

**4 Key Decisions:**

##### 1. **A-based Piano Numbering (A0=1, A4=49)**

**Problem:** Task spec had inconsistent note formula and array alignment.
- Formula: `noteNumber = 12*log2(f/refFreq) + 49`
- Original array: `['C','C#','D',...,'B']` (C-based)
- Issue: A4=49 → (49-1)%12=0 → "C" instead of "A"

**Solution:** Reindexed array to A-based: `['A','A#','B','C','C#','D','D#','E','F','F#','G','G#']`

**Verification:** All 78 tests pass with A4→A, C4→C, B4→B

**Impact:** Zero (localized to tuner frontend)

##### 2. **Transposition: +9 Semitones for Eb (not +3)**

**Problem:** Spec §6.4 says "+3 semitones" but example shows +9 (C4 → Eb3 concert → F#5 display).

**Resolution:** Implemented +9 semitones (musically correct).
- Eb instrument is a major sixth LOWER in pitch than notated
- Therefore, displayed note is major sixth (9 semitones) HIGHER than concert pitch
- Example: Concert A4 → display F#5 (correct)

**Spec Fix Needed:** Hill/Stark should update §6.4 from "+3" to "+9 semitones"

**Impact:** Correct musical behavior established

##### 3. **5th Navigation Tab: "Werkzeuge" (Tools)**

**Problem:** AppShell had 4 tabs; spec mentions Werkzeuge-Tab for Tuner + future tools.

**Solution:** Added 5th StatefulShellBranch with `/app/tuner` route.
- AppShell.dart: +1 NavigationDestination (Icons.tune, "Werkzeuge")
- AppRouter: 5-branch navigation structure

**Future Work:** Wanda (UX) to define additional tools in Werkzeuge tab (Metronom? Stimmblatt-Generator?).

**Impact:** Frontend architecture ready for tool expansion

##### 4. **AudioAnalyzer Interface (No Platform Channel in MS3)**

**Problem:** Platform channels (CoreAudio/Oboe/WASAPI) are complex, per-platform.

**Solution:** Abstract `AudioAnalyzer` interface + `MockAudioAnalyzer` for tests.
- Production code: `audioAnalyzerProvider` overridable
- Tests: Riverpod override with MockAudioAnalyzer
- Vision agent: Will implement PlatformAudioAnalyzer (MS3.5)

**Pattern:** Interface-driven design, mockable for TDD.

**Impact:** Unblocks UI development; audio layer decoupled.
