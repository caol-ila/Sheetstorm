# E2E: Token lifecycle (Storage → Refresh → Expiry)

**Labels:** `milestone:1.5`, `testing`, `e2e`

## Beschreibung

End-to-End-Tests für den JWT-Token-Lebenszyklus: Persistenz, Refresh und Fehlerbehandlung bei Storage-Problemen.

## Testfälle

### Token-Persistenz
- [ ] Nach Login: Token wird im Storage gespeichert
- [ ] Token überlebt Page-Reload (User bleibt eingeloggt)
- [ ] Token überlebt Tab-Wechsel
- [ ] Mehrere Tabs teilen denselben Auth-State

### API-Calls mit gültigem Token
- [ ] Authentifizierte API-Requests enthalten Authorization-Header
- [ ] API-Response wird korrekt verarbeitet
- [ ] Parallele API-Calls funktionieren mit demselben Token

### Token-Refresh bei 401
- [ ] Bei 401-Response: Automatischer Refresh-Token-Request
- [ ] Nach Refresh: Ursprünglicher Request wird wiederholt
- [ ] Neuer Token wird im Storage aktualisiert
- [ ] UI zeigt keine Unterbrechung während Refresh

### Storage-Fehler
- [ ] Graceful Handling wenn localStorage nicht verfügbar
- [ ] Graceful Handling wenn Storage voll ist
- [ ] Graceful Handling bei Corrupt-Token im Storage
- [ ] User wird zum Login weitergeleitet wenn Token und Refresh beide ungültig

## Technische Hinweise

- Playwright `page.evaluate()` für Storage-Manipulation verwenden
- Network-Interception für 401-Simulation: `page.route()` 
- Storage-Clearing zwischen Tests sicherstellen
- Race-Condition bei parallelen 401-Responses testen
